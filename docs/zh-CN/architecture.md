## 高层设计

`winget-source-builder` 是一个基于 Rust 开发的 CLI 工具，包含轻量级原生互操作层。设计优先考虑以下特性：

- **增量处理** — 仅重新处理发生变更的内容
- **状态隔离** — 构建状态独立于 Git 提交历史
- **后端抽象** — 支持多种索引写入策略
- **跨平台** — 兼容 Windows、Linux 和 macOS

### 组件职责

| 组件 | 职责 |
|-----------|----------------|
| **扫描器** | 发现 YAML 清单文件，读取文件内容 |
| **哈希计算器** | 计算 SHA256 哈希值用于变更检测 |
| **合并器** | 按照标准格式合并多文件清单 |
| **差异比较器** | 对比当前状态与存储的历史状态 |
| **状态管理器** | 从 SQLite 中持久化和读取状态 |
| **索引生成器** | 生成最终的源索引（v1 或 v2 版本） |
| **打包器** | 使用平台工具创建 MSIX 文件 |

## 构建流水线

### 阶段 1：扫描

1. 遍历清单仓库的目录树
2. 识别 YAML 文件（`.yaml`、`.yml` 后缀）
3. 按包标识符和版本对文件进行分组
4. 解析每个文件提取元数据

### 阶段 2：哈希计算

为每个文件计算以下哈希值：

- `raw_file_hash` — 原始文件字节的 SHA256 哈希（用于缓存）
- `version_content_sha256` — 标准合并后内容的哈希
- `version_installer_sha256` — 仅影响安装程序的字段的哈希

### 阶段 3：差异比较

将当前哈希值与状态数据库中的值进行对比：

- **未变更** — 跳过重新处理
- **已修改** — 重建对应版本
- **新增** — 添加到状态库
- **已删除** — 从状态库中移除（如果仓库中不存在）

### 阶段 4：合并

针对发生变更的版本：

1. 加载该版本的所有清单文件
2. 遵循 WinGet 优先级规则合并字段
3. 应用标准化处理（例如移除名称中的法律后缀）
4. 验证合并结果

### 阶段 5：索引生成

根据 `--index-version` 参数选择格式：

**v1 格式：**

- 创建内置 SQLite 索引的 `source.msix`
- 生成托管的合并清单

**v2 格式：**

- 创建带优化索引的 `source2.msix`
- 生成 `packages/<id>/<hash>/versionData.mszyml` 附属文件
- 生成托管的合并清单

### 阶段 6：暂存

将所有输出写入 `state-dir/staging/build-<id>/` 目录：

- 支持提交前的校验
- 支持多次发布同一构建版本
- 支持回滚到历史构建

### 阶段 7：发布

在 `publish` 命令中执行以下操作：

1. 复制暂存文件到输出目录
2. 打包 `source.msix` 或 `source2.msix`
3. 对 MSIX 进行签名（如果指定）
4. 更新已发布状态跟踪

## 状态管理

状态存储是一个 SQLite 数据库，维护以下内容：

### 表结构

| 表名 | 用途 |
|-------|---------|
| `files` | 当前文件快照（路径、哈希、修改时间） |
| `versions` | 版本元数据和内容哈希 |
| `packages` | 包级别聚合数据 |
| `builds` | 构建历史，包含时间戳和状态 |
| `published_files` | 已发布输出中的文件清单 |

### 状态流转

```
                    ┌─────────────┐
         ┌─────────▶│  干净状态   │◀────────┐
         │          │             │         │
         │          └─────────────┘         │
         │                  │               │
    存在变更的构建│          │无变更的构建   │清理
         │                  │               │
         ▼                  │               │
┌─────────────────┐         │        ┌─────┴─────┐
│  变更状态       │─────────┘        │  未变更状态 │
│  (文件已修改)   │                  │           │
└─────────────────┘                  └───────────┘
         │
         │发布
         ▼
┌─────────────────┐
│  已发布状态     │
│                 │
└─────────────────┘
```

### 状态隔离

核心原则：构建工具不依赖 Git 历史，仅对比以下内容：

- **当前仓库状态** — 磁盘上当前的文件内容
- **上次发布状态** — SQLite 数据库中存储的内容

这意味着：

- 你可以切换分支、重写历史或修改提交
- 构建工具仅识别当前文件内容
- 不依赖 Git 安装或仓库状态干净

## 哈希模型

构建工具使用多种哈希类型实现不同用途：

### 哈希类型

```
┌─────────────────────────────────────────────────────────────┐
│                     哈希层级结构                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │            raw_file_hash                            │    │
│  │    （原始文件字节的 SHA256 哈希）                    │    │
│  │         用途：扫描缓存                               │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │       version_content_sha256                        │    │
│  │  （合并后的标准清单哈希）                            │    │
│  │       用途：重发布决策                               │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │      version_installer_sha256                       │    │
│  │ （安装程序 URL、类型、架构相关字段的哈希）            │    │
│  │      用途：验证路由                                 │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │     published_manifest_sha256                       │    │
│  │    （最终托管清单字节的哈希）                        │    │
│  │         用途：完整性校验                             │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 内容排除规则

特定字段会被排除在部分哈希计算之外：

| 字段 | 内容哈希 | 安装程序哈希 |
|-------|--------------|----------------|
| `PackageIdentifier` | ✓ | ✓ |
| `PackageVersion` | ✓ | ✓ |
| `InstallerUrl` | ✓ | ✓ |
| `InstallerSha256` | ✓ | ✓ |
| `Commands` | ✓ | ✗ |
| `Protocols` | ✓ | ✗ |
| `FileExtensions` | ✓ | ✗ |

这意味着修改 `Commands` 字段需要重新发布（内容哈希变更），但无需重新验证（安装程序哈希未变更）。

### 哈希示例

**场景 1：修复包描述中的拼写错误**

- `raw_file_hash` — 变更
- `version_content_sha256` — 变更
- `version_installer_sha256` — 未变更
- **结果：** 重新发布清单，无需验证

**场景 2：更新安装程序 URL**

- `raw_file_hash` — 变更
- `version_content_sha256` — 变更
- `version_installer_sha256` — 变更
- **结果：** 重新发布并加入验证队列

**场景 3：文件无变更**

- 所有哈希与存储状态匹配
- **结果：** 完全跳过重新处理

## 输出格式

### v1 格式（ legacy 版本）

```
output/
├── source.msix              # 主索引包
└── manifests/
    └── v/
        └── Vendor/
            └── App/
                └── <hash>/
                    └── manifest.yaml    # 托管合并清单
```

**特点：**

- 单个 `source.msix` 内置 SQLite 数据库
- 所有版本数据都包含在索引中
- 客户端初始下载体积较大

### v2 格式（推荐版本）

```
output/
├── source2.msix             # 主索引包（体积更小）
├── packages/
│   └── Vendor.App/
│       └── <hash8>/
│           └── versionData.mszyml    # 每个版本的元数据
└── manifests/
    └── v/
        └── Vendor/
            └── App/
                └── <hash>/
                    └── manifest.yaml    # 托管合并清单
```

**特点：**

- 初始下载 `source2.msix` 体积更小
- 通过 `versionData.mszyml` 懒加载版本数据
- 更适合包含大量包的仓库
- 需要 WinGet 客户端 1.5 及以上版本

### 格式选择指南

| 场景 | 推荐格式 |
|-----------|-------------------|
| 新仓库 | v2 |
| 现有 v1 用户 | v2（配合迁移指南） |
| 旧版 WinGet 客户端（<1.5） | v1 |
| 大型仓库（1000+ 个包） | v2 |

## 后端抽象

构建工具支持两种索引操作后端：

### Rust 后端（默认）

- 纯 Rust 实现
- 跨平台（Windows、Linux、macOS）
- 直接写入 SQLite 索引
- 推荐所有场景使用

### WinGetUtil 后端

- 使用微软官方 `WinGetUtil.dll`
- 仅支持 Windows
- 与官方 WinGet 最大程度兼容
- 运行时需要 `WinGetUtil.dll`

### 后端选择

```powershell
# 默认（Rust 后端）
winget-source-builder build --backend rust

# Windows 环境下使用官方兼容模式
winget-source-builder build --backend wingetutil
```

除非你特别需要 `WinGetUtil.dll` 的验证兼容性，否则推荐使用 Rust 后端。
