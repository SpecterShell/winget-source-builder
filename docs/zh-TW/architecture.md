## 整體設計

`winget-source-builder` 是一個帶有薄原生互操作邊界的 Rust CLI。設計優先考慮：

- **增量處理** — 只重新處理變更的內容
- **狀態隔離** — 建置狀態獨立於 Git 歷史
- **後端抽象** — 支援多種索引寫入策略
- **跨平台** — 在 Windows、Linux 和 macOS 上執行

### 元件職責

| 元件 | 職責 |
|------|------|
| **掃描器** | 發現 YAML 清單、讀取檔案內容 |
| **雜湊計算器** | 計算 SHA256 雜湊用於變更偵測 |
| **合併器** | 將多檔案清單合併為規範形式 |
| **差異比較器** | 比較目前狀態與儲存狀態 |
| **狀態管理器** | 從 SQLite 持久化和檢索狀態 |
| **索引寫入器** | 產生最終來源索引（v1 或 v2） |
| **封裝器** | 使用平台工具建立 MSIX 檔案 |

## 建置流程

### 第一階段：掃描

1. 遍歷清單倉庫目錄樹
2. 識別 YAML 檔案（`.yaml`、`.yml`）
3. 按套件識別碼和版本分組檔案
4. 解析每個檔案提取中繼資料

### 第二階段：雜湊計算

為每個檔案計算：

- `raw_file_hash` — 原始檔案位元組的 SHA256（用於快取）
- `version_content_sha256` — 規範合併內容的雜湊
- `version_installer_sha256` — 僅影響安裝程式欄位的雜湊

### 第三階段：差異比較

將目前雜湊與狀態資料庫比較：

- **未變更** — 跳過重新處理
- **已修改** — 重建該版本
- **新增** — 新增到狀態
- **已刪除** — 從狀態中移除（如果不在倉庫中）

### 第四階段：合併

對於變更的版本：

1. 載入該版本的所有清單檔案
2. 按 WinGet 的優先順序規則合併欄位
3. 套用規範化（從名稱中移除法律後綴等）
4. 驗證合併結果

### 第五階段：索引產生

基於 `--index-version`：

**v1 格式：**

- 建立含嵌入式 SQLite 索引的 `source.msix`
- 產生託管的合併清單

**v2 格式：**

- 建立含最佳化索引的 `source2.msix`
- 產生 `packages/<識別碼>/<雜湊>/versionData.mszyml` sidecar 檔案
- 產生託管的合併清單

### 第六階段：暫存

將所有輸出寫入 `state-dir/staging/build-<id>/`：

- 這允許在提交前驗證
- 支援多次發佈同一建置
- 支援回復到之前的建置

### 第七階段：發佈

在 `publish` 命令中：

1. 將暫存檔案複製到輸出目錄
2. 封裝 `source.msix` 或 `source2.msix`
3. 簽章 MSIX（如果需要）
4. 更新已發佈狀態追蹤

## 狀態管理

狀態儲存是一個 SQLite 資料庫，維護：

### 資料表

| 資料表 | 用途 |
|-------|------|
| `files` | 目前檔案快照（路徑、雜湊、修改時間） |
| `versions` | 版本中繼資料和內容雜湊 |
| `packages` | 套件級聚合資料 |
| `builds` | 帶時間戳和狀態的建置歷史 |
| `published_files` | 發佈輸出中的檔案清單 |

### 狀態轉換

```
                    ┌─────────────┐
         ┌─────────▶│    乾淨     │◀────────┐
         │          │    狀態     │         │
         │          └─────────────┘         │
         │                  │               │
    建置  │ 有變更           │建置           │清理
         │                  │無變更         │
         ▼                  │               │
┌─────────────────┐         │        ┌─────┴─────┐
│    修改狀態     │─────────┘        │   未變更  │
│   （髒檔案）    │                  │    狀態   │
└─────────────────┘                  └───────────┘
         │
         │發佈
         ▼
┌─────────────────┐
│    已發佈       │
│     狀態        │
└─────────────────┘
```

### 狀態隔離

關鍵原則：建置器不關心 Git 歷史。它比較：

- **目前倉庫狀態** — 磁碟上的目前內容
- **最後發佈狀態** — SQLite 資料庫中的內容

這意味著：

- 你可以切換分支、重寫歷史或修改提交
- 建置器只看到目前檔案內容
- 不依賴 Git 是否存在或乾淨

## 雜湊模型

建置器使用多種雜湊類型用於不同目的：

### 雜湊類型

```
┌─────────────────────────────────────────────────────────────┐
│                        雜湊層級                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │            raw_file_hash                            │    │
│  │         （原始檔案位元組的 SHA256）                   │    │
│  │              用於：掃描快取                          │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │       version_content_sha256                        │    │
│  │       （合併規範清單的雜湊）                          │    │
│  │            用於：重新發佈決策                        │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │      version_installer_sha256                       │    │
│  │    （安裝程式 URL、類型、架構的雜湊）                 │    │
│  │           用於：驗證路由                             │    │
│  └─────────────────────────────────────────────────────┘    │
│                         │                                    │
│                         ▼                                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │     published_manifest_sha256                       │    │
│  │      （最終託管清單位元組的雜湊）                     │    │
│  │           用於：完整性驗證                           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 內容排除

某些欄位從特定雜湊中排除：

| 欄位 | 內容雜湊 | 安裝程式雜湊 |
|------|---------|-------------|
| `PackageIdentifier` | ✓ | ✓ |
| `PackageVersion` | ✓ | ✓ |
| `InstallerUrl` | ✓ | ✓ |
| `InstallerSha256` | ✓ | ✓ |
| `Commands` | ✓ | ✗ |
| `Protocols` | ✓ | ✗ |
| `FileExtensions` | ✓ | ✗ |

這意味著變更 `Commands` 需要重新發佈（內容雜湊變更）但不需要重新驗證（安裝程式雜湊未變）。

### 雜湊範例

**情境 1：修正套件描述中的錯字**

- `raw_file_hash` — 變更
- `version_content_sha256` — 變更
- `version_installer_sha256` — 未變更
- **結果：** 重新發佈清單，無需驗證

**情境 2：更新安裝程式 URL**

- `raw_file_hash` — 變更
- `version_content_sha256` — 變更
- `version_installer_sha256` — 變更
- **結果：** 重新發佈並新增到驗證佇列

**情境 3：檔案無變更**

- 所有雜湊匹配儲存狀態
- **結果：** 完全跳過重新處理

## 輸出格式

### v1 格式（舊版）

```
output/
├── source.msix              # 主索引套件
└── manifests/
    └── v/
        └── Vendor/
            └── App/
                └── <雜湊>/
                    └── manifest.yaml    # 託管的合併清單
```

**特性：**

- 單個含嵌入式 SQLite 的 `source.msix`
- 所有版本資料在索引中
- 用戶端初始下載較大

### v2 格式（推薦）

```
output/
├── source2.msix             # 主索引套件（更小）
├── packages/
│   └── Vendor.App/
│       └── <hash8>/
│           └── versionData.mszyml    # 每版本中繼資料
└── manifests/
    └── v/
        └── Vendor/
            └── App/
                └── <雜湊>/
                    └── manifest.yaml    # 託管的合併清單
```

**特性：**

- 更小的初始 `source2.msix` 下載
- 透過 `versionData.mszyml` 延遲載入版本資料
- 更適合包含大量套件的倉庫
- 需要 WinGet 用戶端 1.5+

### 格式選擇指南

| 情況 | 推薦格式 |
|------|---------|
| 新倉庫 | v2 |
| 現有 v1 使用者 | v2（附移轉指南） |
| 舊版 WinGet 用戶端（< 1.5） | v1 |
| 大型倉庫（1000+ 套件） | v2 |

## 後端抽象

建置器支援兩種後端用於索引操作：

### Rust 後端（預設）

- 純 Rust 實作
- 跨平台（Windows、Linux、macOS）
- 直接 SQLite 索引寫入
- 推薦用於所有情境

### WinGetUtil 後端

- 使用 Microsoft 的 `WinGetUtil.dll`
- 僅 Windows
- 與官方 WinGet 最大相容性
- 執行期需 `WinGetUtil.dll`

### 後端選擇

```powershell
# 預設（Rust）
winget-source-builder build --backend rust

# Windows 官方相容
winget-source-builder build --backend wingetutil
```

除非你需要 `WinGetUtil.dll` 相容性用於驗證目的，否則推薦使用 Rust 後端。
